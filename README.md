# Forensics Detective: PDF Provenance Detection

## Project Overview

**Forensics Detective** is a machine learning-powered digital forensics tool that analyzes PDF documents to determine their creation method. By converting PDFs to binary images and applying image classification techniques, the system can identify whether a PDF was generated by Microsoft Word, Google Docs, Python/ReportLab, LibreOffice, or browser-based print-to-PDF functionality.

### Key Innovation

The project's core innovation lies in converting PDF binary structure into grayscale images, where each byte value (0-255) corresponds to a pixel intensity. This visualization approach transforms abstract binary patterns inherent in different PDF generation engines into visually distinguishable signatures that machine learning models can classify.

### Research Impact

This tool addresses a significant gap in current digital forensics capabilities. While existing expert systems rely heavily on metadata analysis for document provenance, such information can be easily manipulated or stripped by sophisticated actors. The binary visualization approach operates at a deeper structural level, detecting generation signatures that are inherently embedded in the PDF creation process and resistant to simple metadata manipulation.

## Dataset Description

### Source Documents
- **398 Wikipedia articles** converted to Microsoft Word documents (.docx)
- **Content diversity**: Topics span science, history, geography, arts, philosophy, sports
- **Standardized format**: Identical content across all PDF generation methods
- **Filename convention**: Underscored naming for command-line compatibility

### PDF Generation Methods

1. **Microsoft Word** (`word_pdfs/`)
   - Generated using Word's native "Export as PDF" functionality
   - Contains Word-specific creator tool signatures and structural patterns

2. **Google Docs** (`google_docs_pdfs/`)
   - Generated using Google Docs "Download as PDF" functionality
   - Uses Google's cloud-based PDF generation service
   - Identified by Skia/PDF producer signatures

3. **Python/ReportLab** (`python_pdfs/`)
   - Programmatically generated using Python's ReportLab library
   - Shows distinct binary signatures with different intensity patterns
   - Mean intensity: ~73.17 vs ~121 for Word/Google Docs

4. **LibreOffice** (`libreoffice_pdfs/`) - **NEW**
   - Generated using LibreOffice's headless conversion
   - Uses LibreOffice's internal PDF generation engine
   - Provides open-source alternative PDF generation signature

5. **Browser/HTML** (`html_pdfs/`) - **NEW**
   - Generated using Chromium's print-to-PDF functionality
   - Converts DOCX → HTML → PDF via Playwright
   - Represents web-based document generation signatures

## Usage Instructions

### Complete Workflow

The project follows a systematic pipeline from document generation to classification:

#### Step 1: Generate PDFs from Different Sources

**Microsoft Word PDFs:**
```bash
cd src/pdf_conversions
python convert_word_pdfs.py
```
- Requires Microsoft Word installed on macOS
- Uses AppleScript for automated conversion
- Generates PDFs in `data/word_pdfs/`

**Google Docs PDFs:**
```bash
python convert_google_docs_pdfs.py
```
- Requires Google API credentials setup
- Uses multiprocessing for batch conversion
- Generates PDFs in `data/google_docs_pdfs/`

**Python/ReportLab PDFs:**
```bash
python convert_python_pdfs.py
```
- Extracts text from Word documents
- Generates PDFs using ReportLab library
- Generates PDFs in `data/python_pdfs/`

**LibreOffice PDFs (NEW):**
```bash
python convert_libreoffice_pdfs.py
```
- Uses LibreOffice headless mode
- Converts DOCX files to PDF via command line
- Generates PDFs in `data/libreoffice_pdfs/`

**Browser/HTML PDFs (NEW):**
```bash
python convert_html_pdfs.py
```
- Converts DOCX → HTML → PDF using Playwright
- Uses Chromium's print-to-PDF functionality
- Generates PDFs in `data/html_pdfs/`

#### Step 2: Convert PDFs to Binary Images

```bash
cd src
python image_conversion.py
```

This script:
- Reads PDF binary data
- Converts bytes to grayscale pixel values
- Creates square-ish images for consistent dimensions
- Saves PNG images in `data/converted_pngs/`

#### Step 3: Train Classification Models

**Baseline 2-Class Classification:**
```bash
cd src/classifications
python train_baseline_classifiers.py
```
- Trains SVM and SGD on Word vs Google Docs
- Establishes baseline performance metrics

**3-Class Classification:**
```bash
python train_3class_classifiers.py
```
- Adds Python/ReportLab as third class
- Analyzes class separability

**5-Class Classification:**
```bash
python train_5class_classifiers.py
```
- Includes all five PDF generation methods
- Implements hyperparameter tuning with cross-validation
- Trains SVM, SGD, RandomForest, HistGradientBoosting, and CNN

#### Step 4: Analyze Results

Results are automatically saved to the `results/` directory:
- `performance_metrics.csv`: Accuracy, precision, recall, F1 scores
- `confusion_matrices/`: Detailed confusion matrices for each model
- `comparison_images/`: Visual comparisons of different PDF types
- Trained model files (`.pkl`, `.h5`, `.keras`)

### Individual Script Usage

#### LibreOffice PDF Conversion (`convert_libreoffice_pdfs.py`)

This script converts Word documents to PDF using LibreOffice's headless mode:

**Key Features:**
- Uses LibreOffice command-line interface
- Processes files in batch with progress tracking
- Skips already converted files
- Handles conversion errors gracefully

**Usage:**
```bash
python convert_libreoffice_pdfs.py
```

**Technical Details:**
- Path: `/Applications/LibreOffice.app/Contents/MacOS/soffice`
- Command: `--headless --convert-to pdf --outdir`
- Output: PDFs in `data/libreoffice_pdfs/`

#### HTML/Browser PDF Conversion (`convert_html_pdfs.py`)

This script converts Word documents to PDF via HTML using Playwright:

**Key Features:**
- Converts DOCX to HTML using mammoth library
- Renders HTML to PDF using Chromium
- Preserves formatting and images
- Uses base64 encoding for inline images

**Usage:**
```bash
python convert_html_pdfs.py
```

**Technical Details:**
- DOCX → HTML conversion via mammoth
- HTML → PDF via Playwright Chromium
- Custom CSS for print formatting
- Output: PDFs in `data/html_pdfs/`

**Process Flow:**
1. Read DOCX file using mammoth
2. Convert to HTML with inline images
3. Wrap in full HTML document with CSS
4. Load in Chromium browser
5. Generate PDF using print functionality

## Model Performance

### Current Results (5-Class Classification)

| Model | Accuracy | Precision (Macro) | Recall (Macro) | F1 (Macro) |
|-------|----------|-------------------|----------------|------------|
| SVM | 91.0% | 92.5% | 91.0% | 91.0% |
| SGD | 91.0% | 91.8% | 91.0% | 91.1% |
| RandomForest | 90.0% | 90.6% | 90.0% | 89.9% |
| HistGradientBoosting+PCA | 89.0% | 91.0% | 89.0% | 89.0% |
| CNN | 71.0% | 62.1% | 71.0% | 64.8% |

### Key Findings

1. **Class Separability**: Python-generated PDFs show distinct binary signatures (mean intensity: 73.17 vs ~121 for Word/Google)
2. **Model Performance**: SVM and SGD achieve highest accuracy (~91%)
3. **Scalability**: Performance maintained across different document lengths and content types
4. **Robustness**: Models resistant to metadata stripping and basic obfuscation

## Advanced Features

### Hyperparameter Tuning
- **SVM**: Grid search with RBF/linear kernels, C values, gamma parameters
- **SGD**: Randomized search with different loss functions and penalties
- **RandomForest**: Grid search with n_estimators (200, 400) and max_depth (None, 20, 40)
- **HistGradientBoosting+PCA**: Randomized search with learning_rate (0.05, 0.1), max_leaf_nodes (15, 31), max_iter (80, 120, 160), l2_regularization (0.0, 1e-4, 1e-3), min_samples_leaf (20, 50), and PCA components (128, 256)
- **CNN**: Architecture optimization with dropout, batch normalization

### Cross-Validation
- 5-fold cross-validation for robust performance estimation
- Stratified splits to maintain class distribution
- Macro-averaged metrics for balanced evaluation

### Model Persistence
- Trained models saved as `.pkl` files
- CNN models saved in multiple formats (`.h5`, `.keras`)
- Feature scalers preserved for consistent preprocessing

## Troubleshooting

### Common Issues

**LibreOffice Conversion Fails:**
- Ensure LibreOffice is installed and accessible
- Check file permissions for input/output directories
- Verify DOCX files are not corrupted

**Google API Errors:**
- Refresh OAuth tokens if expired
- Check API quotas and limits
- Verify credentials file format

**Memory Issues:**
- Reduce `max_samples_per_class` parameter
- Use smaller `target_size` for images
- Process files in smaller batches

**CNN Training Issues:**
- Reduce batch size if GPU memory insufficient
- Increase patience for early stopping
- Check class balance and weights

### Performance Optimization

**For Large Datasets:**
- Use multiprocessing for PDF conversion
- Implement batch processing for image conversion
- Consider PCA for dimensionality reduction

**For Faster Training:**
- Use smaller image dimensions (e.g., 100x100)
- Reduce number of samples per class for testing
- Enable parallel processing with `n_jobs=-1`
